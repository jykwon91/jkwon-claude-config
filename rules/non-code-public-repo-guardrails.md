# Non-Code Public Repo Guardrails

Public repos have strong guardrails for **code** (PR review, CI, secret scanning, never-commit-`.env`). They have weak or no guardrails for **non-code content** — roadmaps, internal plans, design docs, audit reports, anything generated by tools like the memory curator.

Non-code content can leak more strategic info than code does: roadmaps reveal what you're building and when, design docs reveal architectural reasoning, internal context can leak personal info, infrastructure paths, or competitive direction.

## The rule

Before committing or recommending a push of any non-code file to a **public** repo:

1. **Does the file describe future intent?** — Roadmaps, "soon" lists, deferred decisions, planning docs, sprint plans → keep local.
2. **Does it reference VPS / infrastructure paths?** — `/srv/...`, IPs, SSH user/host, deploy targets → reveals attack surface; keep local.
3. **Does it contain personal context about the operator?** — "user is actively job-hunting", health/family/financial, schedule, location → keep local.
4. **Does it cross-reference private files?** — Links to `~/.claude/projects/.../memory/*.md`, paths inside `~/.claude/` → breaks for others; signals private context exists; keep local.
5. **Does it expose competitive direction or strategic decisions in flux?** — "should we use X or Y", pivoting thoughts, unreleased features, pricing → keep local.
6. **Would another developer using this same repo benefit from this content?** — If no, it's `personal`-tier per `docs/change-types.md`; never commit.

If ANY of (1)-(5) is yes, the file is `personal`-tier. Don't commit to public main.

## What's still safe for public repos

- Code (obvious)
- API documentation reflecting actual behavior
- Public READMEs and CHANGELOGs
- Architecture docs describing shipped state (not "we're considering...")
- Privacy Policy, Terms of Service, contributor guides
- Public PR descriptions and commit messages — within reason; avoid leaking internal reasoning

## How to handle local-only artifacts

```bash
# Add to .gitignore
echo "ROADMAP.md" >> .gitignore
echo "MEMORY_AUDIT_*.md" >> .gitignore
echo "*.scratch.md" >> .gitignore

# If already committed by mistake
git rm --cached <file>
git commit -m "chore: untrack <file> — personal artifact"

# Better location for cross-session personal artifacts:
# ~/.claude/projects/<project-hash>/<artifact>.md
```

The memory curator pipeline (`g-pipeline-curate-memory`) should write ROADMAP.md to `~/.claude/projects/<project-hash>/ROADMAP.md`, not to the project working tree.

## Self-honest check

When you generate any non-code artifact and feel "this should go in the repo so it's preserved" — stop. Ask: "Does it contain anything from items 1-5?" If yes, preservation goes to `~/.claude/`, not public main.

**Default for any non-code file you didn't explicitly write as public-facing: local-only.** Public commits opt-in, not opt-out.
