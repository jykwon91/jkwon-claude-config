# Non-Code Public Repo Guardrails

Public repos already have strong guardrails for **code**: PR review, CI, secret scanning, dependency review, never-commit-`.env`. They have weak or no guardrails for **non-code content** — roadmaps, internal plans, design docs, audit reports, retrospectives, anything generated by tools like the memory curator.

Non-code content can leak more strategic information than code does. A roadmap reveals what you're building and when. A design doc reveals architectural reasoning. An internal context note can leak personal info, infrastructure details, or competitive direction. None of this triggers existing review gates.

## The rule

Before committing or recommending a push of any non-code file to a **public** repo, run this checklist:

1. **Does the file describe future intent?** — Roadmaps, "soon" lists, deferred decisions, planning docs, sprint plans. → Default: keep local. Operator-only artifact.
2. **Does it reference VPS / infrastructure paths?** — `/srv/...`, IP addresses, SSH user/host, deploy targets. → Reveals attack surface. Keep local.
3. **Does it contain personal context about the operator?** — "user is actively job-hunting", health/family/financial info, working schedule, location. → Strictly personal. Keep local.
4. **Does it cross-reference private files?** — Links to `~/.claude/projects/.../memory/*.md`, paths inside `~/.claude/`, internal scratch directories. → Will break for anyone else who clones; signals private context exists. Keep local.
5. **Does it expose competitive direction or strategic decisions in flux?** — "should we use X or Y", "we're considering pivoting to Z", unreleased features, pricing thoughts. → Strategy artifact. Keep local.
6. **Would another developer using this same repo benefit from this content?** — If the answer is no, it's a personal artifact. Apply `personal`-tier per `docs/change-types.md` — never commit to a shared repo.

If ANY of (1)-(5) is yes, the file is `personal`-tier per the change-types framework. Don't commit to public main.

## What's still safe

These ARE appropriate for public repo commits:
- Code (obvious)
- API documentation reflecting actual code behavior
- Public-facing READMEs and CHANGELOGs
- Architecture docs that describe shipped state (not "we're considering...")
- Privacy Policy, Terms of Service, contributor guides
- Public PR descriptions and commit messages — within reason; avoid leaking internal reasoning

## How to handle local-only artifacts

For files that should exist for the operator but never enter the public repo:

```bash
# Add to .gitignore
echo "ROADMAP.md" >> .gitignore
echo "MEMORY_AUDIT_*.md" >> .gitignore
echo "*.scratch.md" >> .gitignore   # generic catch for ad-hoc planning files

# If the file was already committed by mistake
git rm --cached <file>
git commit -m "chore: untrack <file> — personal artifact, see rules/non-code-public-repo-guardrails"

# Better location for cross-session personal artifacts:
# ~/.claude/projects/<project-hash>/<artifact>.md
# (lives next to auto-memory, never in any working tree)
```

The memory curator pipeline (`g-pipeline-curate-memory`) writes ROADMAP.md to the project working tree by default. After this rule lands, the curator should be updated to write to `~/.claude/projects/<project-hash>/ROADMAP.md` instead — leaving the working tree alone.

## Failure mode this prevents

Tonight (2026-04-26), the curator pipeline emitted ROADMAP.md to both MyBookkeeper and MyFreeApps working trees. I (the assistant) recommended committing AND pushing them as documentation. The user caught it. The roadmaps contained:

- VPS paths (`/srv/mybookkeeper`)
- "user is actively job-hunting, so MyJobHunter is the priority"
- Cross-references to `~/.claude/projects/.../memory/*.md`
- Open architectural questions about cutover DB topology
- Recently-completed PR lists exposing internal review patterns

All of that benefits the operator's planning. None of it should be in a public repo.

## Relationship to other rules

- **`rules/never-auto-merge-config-repo.md`** — config repo gets stricter treatment (no auto-merge of any kind)
- **`docs/change-types.md`** — defines the `personal` tier this rule enforces
- **`rules/g-auto-capture.md`** — only writes to `stacks/*.md` (public, generic) — already safe
- **`rules/auto-memory-curation.md`** — writes to private `~/.claude/projects/...` — already safe

## Self-honest check

When you generate any non-code artifact and feel the impulse "this should go in the repo so it's preserved" — stop. Ask: "Does it contain anything from items 1-5 above?" If yes, the right preservation location is `~/.claude/`, not public main.

The default for ANY non-code file you didn't explicitly write as public-facing should be: **local-only.** Public commits opt-in, not opt-out.
